# Lecture 2: Document Processing, Data Versioning, and Chatbot Shell

## 1. Purpose of this lecture

Lecture 2 extends the Lecture 1 foundation into a beginner-friendly document-processing project.

The goal is not to build a full RAG application yet. The goal is to prepare document data in a production-style way: keep raw and processed data separate, validate documents before processing, clean text, create chunks, attach metadata, record lineage, and prepare the repository for DVC-based data versioning.

The lecture also adds a simple Streamlit chatbot shell so learners can see how processed knowledge-base work will eventually connect to an application.

## 2. What we built

We added:

- sample raw policy documents
- document loader
- document validation
- text cleaner
- fixed-size chunker with overlap
- processed chunk artifact
- lineage metadata artifact
- DVC preparation
- Streamlit chatbot shell
- rule-based keyword lookup service
- tests for the new document and chat logic

## 3. Why this is still not RAG

The chatbot is not powered by an LLM.

It does not use embeddings, vector databases, semantic search, Azure AI Search, Azure OpenAI, LangChain, LlamaIndex, or any paid API.

The app uses only local keyword matching. This keeps Lecture 2 focused on data preparation, quality checks, artifact generation, lineage, and versioning. Semantic retrieval and generated answers will be introduced later after the document foundation is clear.

## 4. Key concepts demonstrated

### 4.1 Raw versus processed data

Raw documents live under:

```text
data/raw/policies_v1/
```

These are the source documents for the knowledge base. They should be treated as input data.

Processed artifacts live under:

```text
data/processed/
```

These are generated by the document pipeline and can be recreated from the raw documents and code.

Metadata artifacts live under:

```text
data/metadata/
```

These explain how processed files were created.

### 4.2 Document loading

`src/ingestion/document_loader.py` loads supported UTF-8 text files from the configured raw folder.

Each loaded document includes:

- `source_file`
- `source_path`
- `document_version`
- `file_extension`
- `text`

This metadata travels with the document so downstream steps know where every chunk came from.

### 4.3 Document validation

`src/ingestion/validate_documents.py` checks document quality before processing.

Validation checks include:

- input folder exists
- at least one supported document exists
- file extension is supported
- text is not empty
- text meets the minimum length
- source filenames are not duplicated
- required metadata fields exist

If validation fails, the pipeline raises a clear error and does not create processed artifacts.

### 4.4 Text cleaning

`src/processing/text_cleaner.py` keeps cleaning intentionally simple:

- trims leading and trailing whitespace
- normalizes repeated spaces
- reduces excessive blank lines
- preserves paragraph readability

No NLP framework, stemming, lemmatization, OCR, PDF parsing, or table parsing is used.

### 4.5 Fixed-size chunking with overlap

`src/processing/text_chunker.py` implements one strategy:

```text
Fixed-size character chunking with overlap
```

The current config uses:

```yaml
chunk_size: 500
chunk_overlap: 50
```

The code validates that:

- chunk size is positive
- overlap is zero or positive
- overlap is smaller than chunk size

Chunk IDs are deterministic and readable, for example:

```text
leave_policy_v1_chunk_001
```

### 4.6 Metadata and lineage

Each chunk records where it came from:

- source file
- source path
- document version
- chunk index
- chunking strategy
- chunk size
- chunk overlap

The lineage file records the pipeline-level context:

- pipeline name and version
- source version
- input path
- output paths
- validation result
- processed files
- processing steps
- generation timestamp in UTC

### 4.7 DVC preparation

Lecture 2 prepares for DVC, but does not fake DVC-generated files.

The repository includes `.dvcignore` and installs `dvc` through `requirements.txt`. Learners run the DVC commands themselves so they can see how DVC creates `.dvc/`, pointer files, and data ignore rules.

### 4.8 Rule-based chatbot shell

`src/chat_service.py` maps simple keywords to local policy files.

Example mapping:

| Keyword | Source file |
| ------- | ----------- |
| `leave` | `leave_policy.txt` |
| `sick` | `leave_policy.txt` |
| `travel` | `travel_policy.txt` |
| `hotel` | `travel_policy.txt` |
| `cab` | `travel_policy.txt` |
| `reimbursement` | `travel_policy.txt` |
| `laptop` | `it_support_faq.txt` |
| `password` | `it_support_faq.txt` |
| `helpdesk` | `it_support_faq.txt` |

The service returns:

- answer text
- source file
- retrieval method
- `is_llm_response: false`

This makes the boundary clear: the app is a local lookup demo, not an LLM system.

## 8. DVC learner workflow

### 8.1 Initialize DVC

```bash
dvc init
git add .dvc .dvcignore
git commit -m "Initialize DVC for data versioning"
```

### 8.2 Track raw policy documents

```bash
dvc add data/raw/policies_v1
git add data/raw/policies_v1.dvc data/raw/.gitignore
git commit -m "Track policy document version 1 with DVC"
```

### 8.3 Git versus DVC

| Tool | Tracks |
| ---- | ------ |
| Git | Code, configs, tests, docs, `.dvc` metadata |
| DVC | Large data and document snapshots |

### 8.4 What happens internally

`dvc add` calculates a content hash for the data folder. DVC manages the actual raw data snapshot. Git tracks the small `.dvc` pointer file.

A changed document produces a changed DVC hash. The updated `.dvc` pointer file should be committed to Git.

Together, the Git commit and DVC pointer identify the exact code and data state.

### 8.5 Demonstrate a second data version

1. Change one policy document.
2. Run:

```bash
dvc add data/raw/policies_v1
git diff data/raw/policies_v1.dvc
```

3. Observe the changed hash.
4. Commit the updated pointer file.

### 8.6 Restore a data version

```bash
git checkout <commit-id>
dvc checkout
```

`git checkout` restores the code and `.dvc` pointer file for the selected commit. `dvc checkout` restores the data files that match that pointer.

### 8.7 Lecture 2 DVC scope

Lecture 2 does not configure a DVC remote.

That means:

- no `dvc push`
- no `dvc pull`
- no DVC pipelines
- no DVC experiments

Remote storage and automated DVC workflows can be introduced later.

## 9. Local to Azure mapping

| Local artifact | Azure service later |
| -------------- | ------------------- |
| `data/raw/` | Azure Blob Storage raw container |

Blob Storage stores files. Azure AI Search makes processed content searchable. Azure OpenAI embeddings convert text into vectors. Azure OpenAI chat generates answers.

Lecture 2 only explains this mapping. It does not call Azure services.

## 11. Common errors

| Error | Fix |
| ----- | --- |
| Virtual environment not activated | Activate `.venv` before running commands. |
| Missing dependency | Run `pip install -r requirements.txt`. |
| Import-path errors | Run commands from the repository root. |
| Invalid YAML | Check indentation in `configs/config.yaml`. |
| Unsupported file type | Use `.txt` files in Lecture 2. |
| Empty document | Add meaningful text before running the pipeline. |
| Too-short document | Add enough content to meet `minimum_document_length`. |
| Overlap greater than chunk size | Set `chunk_overlap` smaller than `chunk_size`. |
| Streamlit command not found | Install dependencies and activate the virtual environment. |
| DVC not initialized | Run `dvc init` before `dvc add`. |
| Raw data accidentally staged in Git | Unstage it, run `dvc add`, and commit the `.dvc` pointer file instead. |
| Processed artifact missing | Read the validation error, fix the document, and rerun the pipeline. |
